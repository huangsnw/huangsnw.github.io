单点登录

单点登录的概念

“单点登录”的概念是我刚工作不久leader给我提出要做的一个功能需求。我第一反应是懵的，我不知道这个概念是什么，于是赶紧去网上捞了一下。

经过一番搜索，我有了对单点登录的理解：在分布式的系统中，一个服务登录，其他服务都能免登录的功能。

为什么需要“单点登录”

浏览器使用的HTTP协议是无状态的，服务器并不能通过HTTP的访问认出你。所以就需要保持登录状态，其核心思路就是存储用户登录信息。为了避免每次请求都携带用户名和密码，就出现了cookie和session。

cookie

浏览区向服务器发送HTTP请求，服务器执行set-cookie操作，将key和value存储到cookie中，返回给浏览器。浏览器接收到cookie之后保存，之后每次访问都携带上cookie访问服务器。

⚠️key和value是保存在浏览器上的，不安全。

session

浏览器想服务器发送HTTP请求，服务器生成session-id、session到期时间等信息，保存到浏览器内存中，再将session-id、session到期时间通过set-cookie的方式发送给浏览器，浏览器保存cookie，之后每次访问都携带上cookie访问服务器。

token

session存储在服务器，会造成冗余存储，于是就出现了jwt。jwt只需要在服务器上保存一个密钥，然后将生成的jwt发送给浏览器，之后每次访问都携带上jwt访问服务器。

jwt格式：hearder.payload.singature

实现方式

现在在网上看到的大部分实现“单点登录”的方式主要有三种：（1）session；（2）cookies+redis；（3）token

session

在浏览器访问服务器的时候，服务器会在内存中开辟一块内存，用于保存一些用户的数据，这个就是session。这块内存信息是和浏览器关联的，换个浏览器访问就不是原来的数据了。

session的生命周期

（1）浏览器访问服务器创建session；（2）session存活周期到了、服务器非正常关闭session销毁。

⚠️服务器正常关闭，不会清除session，而是会序列化到磁盘上。

session实现“单点登录”的方式

用户访问某个服务之后，服务器存储session，然后通过session的广播机制，将session的内容复制到其他的服务器上。

这样的实现方式会占用服务器内存，造成数据冗余存储。

cookie+redis

登录将cookie存入redis中，以后每次登录都将浏览器携带的cookie和redis中的数据进行对比，如果在 redis 中能查询到该用户相应的登录信息，就说明该用户已登录，就不需要用户进行重复登录了。

token

用户在项目的某个模块进行登录后，系统会按照一定的规则生成字符串，把用户登录之后的信息包含到这个生成的字符串中，然后系统可以将这个字符串返回。这样用户在访问其他的模块时，每次访问的地址栏都会带着生成的字符串（或者 cookie 中带着生成的字符串），被访问模块就可以获取地址栏中的生成字符串（或者获取 cookie 中的生成字符串），然后根据字符串获取用户信息，如果可以获取到用户的登录信息，说明该用户已登录，用户就不需要重复登录了。
